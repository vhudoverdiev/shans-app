# Shans

Flask веб-приложение для личного управления данными.

## Возможности

- Авторизация пользователей
- Учёт бюджета
- История баланса
- Экспорт бюджета в Excel
- Учёт выполненных работ по автомобилю
- Учёт планируемых работ по автомобилю
- Уведомления по обслуживанию автомобиля
- Раздел фотопроектов: город/адрес, временные окна и записи с оплатой

## Технологии

- Python
- Flask
- Flask-Login
- SQLite
- openpyxl
- Gunicorn
- Nginx
- systemd

## Запуск локально

### 1. Клонировать репозиторий

```bash
git clone https://github.com/vhudoverdiev/shans-app.git
cd shans-app
```

## Уведомления

- В проекте используется только встроенный планировщик VK-уведомлений в приложении (ежедневно в 20:00 по Москве).
- Отдельный legacy-скрипт ночной отправки уведомления "Задачи на завтра (...)" удалён из репозитория.
- Дефолтные значения для VK берутся из переменных окружения `VK_ACCESS_TOKEN`, `VK_PROFILE_URL`, `VK_TIMEZONE` только при первичной инициализации БД.
- Для диагностики на сервере используйте CLI-команду:

```bash
flask --app app.py vk-diagnose
flask --app app.py vk-diagnose --remote-check
flask --app app.py vk-set-config --token 'vk1.a.xxxxx' --profile-url 'https://vk.com/your_profile' --timezone 'Europe/Moscow' --enabled
flask --app app.py vk-send-test
```

`--remote-check` дополнительно проверяет токен/профиль через VK API и помогает быстро понять, что сломано (токен, ссылка профиля, время отправки, lock за текущий день).
`vk-set-config` обновляет токен/профиль/таймзону прямо в БД (это основной способ заменить токен на сервере).
`vk-send-test` принудительно отправляет тестовое сообщение "Задачи на завтра" сразу (без ожидания 20:00).

Частые ошибки при `vk-send-test`:
- `VK API error 38` — обычно токен больше не валиден для приложения/контекста. Сгенерируйте новый токен сообщества и обновите `VK_ACCESS_TOKEN`.
- `VK API error 5` — ошибка авторизации токена.
- `VK API error 901` — пользователь не разрешил сообщения от сообщества (нужно написать сообществу первым).

## Деплой на сервер (main)

Если на сервере нет `deploy.sh`, значит локальный серверный репозиторий ещё не подтянут до коммита, где он добавлен.

```bash
cd /var/www/shans-app
git fetch origin
git checkout main
git reset --hard origin/main
```

Проверка, что скрипт появился:

```bash
ls -l /var/www/shans-app/deploy.sh
```

Запуск деплоя:

```bash
cd /var/www/shans-app
chmod +x deploy.sh
./deploy.sh
```

По умолчанию скрипт не удаляет `.env` (режим `CLEAN_MODE=safe`).
Также по умолчанию включена проверка чистоты tracked-файлов после деплоя (`VERIFY_CLEAN=true`).

Если после рестарта сервиса снова появляются изменения в `git status`, это признак того, что в репозитории по ошибке отслеживаются runtime-файлы (например, `__pycache__`, `*.pyc`, `logs/*`).
